# DETAILS FOR UPGRADE from llama.cpp sha `615212` (b4531) to `305ba519` (b10076)

Upgrade date: 2026-07-21
Upstream: `ggml-org/llama.cpp` tag `b10076` = `305ba519ab61cdff8044922cba2347826a04453f`

## Why this upgrade

Aerni, Fluck & Becker published *"On-chain LLM inference under instruction budgets"* on the
DFINITY forum, measuring 10 tok/call on llama_cpp_canister v0.10.1 vs 29 tok/call on their
fork, same Qwen 2.5 0.5B Q8_0 GGUF and same 40B instruction cap.

Root cause confirmed locally: at b4531, `ggml_vec_dot_q8_0_q8_0` in
`ggml/src/ggml-cpu/ggml-cpu-quants.c` had NEON and AVX2 branches but **no
`__wasm_simd128__` branch**, so it fell through to the scalar `#else`. Only
`ggml_vec_dot_q5_0_q8_0` and `ggml_vec_dot_q5_1_q8_1` were vectorized for WASM. Since
matmul is ~98.7% of decode cost, a Q8_0 model ran its dominant kernel without a
hand-written SIMD kernel.

Upstream b10076 ships `ggml/src/ggml-cpu/arch/wasm/quants.c` with 10 WASM SIMD kernels —
q4_0, q4_1, q5_0, q5_1, **q8_0**, q2_K..q6_K. `ggml_vec_dot_q8_0_q8_0` is at line 616.

**Caveat on the 2.9x figure.** Their build measures 6 tok/call scalar and 30 with SIMD on
the *same* build. Our 10 sits above that scalar floor, so the published 10-vs-29 is a
whole-stack delta, not a kernel delta — it also contains 18 months of unrelated upstream
change. Do not claim the kernel alone accounts for it.

### Debugging technique: the faithful wasmtime harness (the most reusable tool here)

The IC gives NO wasm backtrace for an install/runtime trap — only `[TRAP]: heap out of
bounds`. wasmtime does, **if you run the wasm faithfully**. This is what pinned every
wasm-only bug in this upgrade (`scripts/wasm_harness.py`):

- **Stub the `ic0` imports, but back stable memory with a real Python `bytearray`** — not
  no-op zeros. The ic-wasi-polyfill then runs its real BTreeMap / stable-fs init against it,
  so the wasm behaves as on the IC. (Early on I stubbed stable memory as no-ops and got a
  misleading allocator assert; the bytearray backing removed that artifact.)
- **Run the `before_opt` wasm to get NAMED backtraces.** binaryen's `optimize()` strips the
  wasm name section. `build/llama_cpp_before_opt.wasm` (post-wasi2ic, pre-binaryen) keeps it,
  and wasmtime does not care that it still exports globals the IC would reject. This is what
  turned `<wasm function 4240>` into `getenv <- tty_can_use_colors() <- common_params_parse`
  (Bug #5).
- **Call the actual canister methods**, not just instantiate: provide the candid arg via
  `msg_arg_data_size`/`msg_arg_data_copy`, force `is_controller` -> 1 for admin-gated
  endpoints, capture `debug_print`. Plain instantiation only runs the ctors (found Bug #1);
  the load-path getenv fault (Bug #5) needed a real `load_model` call.

**Interpretation rule (the lesson that would have saved the most time):** *if the faithful
harness says the binary is clean but the IC traps, suspect the DEPLOY PIPELINE, not the
binary.* That is exactly the dfx wasm-cache confound — the harness was right; dfx was serving
a stale `.dfx/` copy.

### Recommended porting order for the NEXT upgrade

1. **Native** (`make all-tests` / MockIC) — fastest loop; catches API/merge/link errors. But
   it has real mmap, threads, exceptions, stack and getenv, so it proves almost nothing about
   canister *runtime* behavior (it passed 111/111 while 5 wasm-only bugs sat live).
2. **Faithful wasmtime harness — BEFORE touching dfx:**
   - (a) on `before_opt` (named backtraces): instantiate (runs ctors → catches static-init
     faults), then call `load_model` and `run_update`. getenv-class faults surface at
     arg-parse with a real backtrace, no model needed. Upload a tiny model into the harness
     vFS via `file_upload_chunk` to also exercise the mmap/warmup/decode path.
   - (b) on the optimized `build/llama_cpp.wasm`: confirm `optimize()` did not change behavior
     vs `before_opt`. (The reference fork's `optimize_wasm.py:5` claims "wasm-opt corrupts the
     code" and avoids it. That is about the `wasm-opt` CLI with real optimization passes, NOT
     our `binaryen.py module.optimize(shrink=0, opt=0)`, which we validated end-to-end this
     upgrade — the optimized binary installs, loads a model and generates tokens correctly. So
     this is a cheap sanity check, not a known hazard for our pipeline.)
3. **Local IC replica** — confirmation, not primary debugging. ALWAYS `dfx deploy` or
   `dfx canister install --wasm <path>` (never trust the `.dfx` cache); verify the module hash
   changed; run the full pipeline (upload → load_model → run_update).
4. **Mainnet** — throughput measurement under the 40B cap (the point of the upgrade).

Had this order been followed, step 2a would have caught bugs #1/#5 (and, with a model
uploaded, #3/#4) with named backtraces in minutes each, and step 3 would have been a quick
confirmation instead of a multi-hour cache-confound chase.

## NEW STRATEGY: WASI shims instead of per-file patching

> **Attribution.** This whole strategy — header-shadowing `wasi-shims/` for the threading
> headers, `-fexceptions` + exception/dl stubs so upstream compiles unmodified, instead of
> hand-patching every file — was **learned from the reference fork** (Julien Aerni, Siméon
> Fluck, Dustin Becker, `instruction-bounded-inference-artifact`, MIT). Their build proved the
> approach works on ICP; we adopted and adapted it (e.g. `__cxa_throw` traps via `IC_API::trap`
> rather than `abort()`; the `getenv` override; the per-file guard/axis analysis). Full
> credits in the Attribution section at the end.

Upgrades 0001 and 0002 hand-ported every upstream file: `throw` -> `IC_API::trap`,
outcomment try/catch, strip threading/signals/`getenv`. That cost scales with file count,
and b10076 roughly doubled the source list (~35 -> ~180 TUs incl. `src/models/*.cpp`).

This upgrade instead uses:

- **`wasi-shims/`** (new, repo root) — placed FIRST on `cpp_include_dirs`. Supplies no-op
  `<mutex>`, `<thread>`, `<future>`, `<condition_variable>` plus `__mutex/*.h` sub-header
  interceptors. Each shim `#define`s the libc++ include guards it replaces
  (`_LIBCPP_MUTEX`, `_LIBCPP___MUTEX_ONCE_FLAG_H`, ...) so transitive includes also resolve
  here.
- **`-fexceptions`** in `cpp_compile_flags` and `cpp_link_flags` — overrides icpp-pro's
  default `-fno-exceptions` (`icpp-pro/src/icpp/config_default.py:127`; later flag wins).
- **`src/wasi-exception-stubs.cpp`** (new) — `__cxa_allocate_exception`, `__cxa_free_exception`,
  `__cxa_throw`, `__cxa_begin_catch`, `__cxa_end_catch`, `__cxa_rethrow`.
- **`src/wasi-dl-stubs.cpp`** (new) — `dlopen`/`dlsym`/`dlclose`/`dlerror` no-ops.
- **wasi-emulated libs** — `-lwasi-emulated-{pthread,getpid,signal,process-clocks}` plus the
  matching `-D_WASI_EMULATED_*` defines.

### IMPORTANT: what the shims do NOT do

There is no unwind runtime. `__cxa_throw` **traps the canister** via `IC_API::trap()` and
`__cxa_begin_catch`/`__cxa_end_catch` are no-ops, so **upstream `try`/`catch` never
recovers**. Any upstream catch that used to recover now traps. Code paths that throw during
normal operation must still be guarded with `#ifdef __wasi__`.

Improvement over the reference implementation: theirs calls `abort()`, losing the message.
Ours builds a diagnostic from `type_info::name()` — and, when the mangled name shows a
`std::` type (leading `St`), from `what()` — then calls `IC_API::trap()`. Callers get a real
error message instead of a bare canister trap. Reading `what()` is gated on the type check
because dereferencing an arbitrary thrown object through a vtable is unsafe.

### Stub obligations — a structural lesson for future upgrades

Excluding a `common/*.cpp` for the globals budget does NOT remove the dependency; it converts
it into a **stub obligation**. `common/arg.cpp` reaches into `chat.h`, `download.h`,
`speculative.h` and `json-schema-to-grammar.h` purely to parse and document CLI options, so
their symbols must still link. This is why the stub set could not be enumerated upfront and
grew across C3 iterations.

Three stub files, each with a deliberate behavior split rather than blanket no-ops:

| File | Traps (loud failure) | Returns benign |
|---|---|---|
| `src/wasi-download-stubs.cpp` | network + Docker actions | HF-cache queries (empty is truthful in a canister) |
| `src/wasi-speculative-stubs.cpp` | — | help-text strings, empty type lists |
| `src/wasi-chat-stubs.cpp` | grammar construction, non-`none` reasoning formats | `resolve_refs`, `resolves_to_string` |

Rationale for trapping on grammars: silently returning an empty grammar would let a
`--grammar` request produce UNCONSTRAINED output that looks successful. A trap names the cause.

### GOTCHA: header shadowing in `src/`

Our wrapper has its own `src/download.h`, and icpp-pro puts `src/` on the include path. A
stub file living in `src/` that does `#include "download.h"` resolves to **our** canister
header, not llama.cpp's — it compiles against entirely the wrong declarations. Always use
explicit `common/`-prefixed includes in the stub files (`#include "common/download.h"`).
We also have `src/upload.h`; watch for any future collision.

### `params.use_jinja` — a tokenization trap

b10076 defaults `use_jinja` to **true** (`common/common.h:633`) and computes
`add_bos = llama_vocab_get_add_bos(vocab) && !params.use_jinja`. Left alone that silently
flips `add_bos` to false and **changes tokenization**.

Fix: force `params.use_jinja = false` right after `common_params_parse` (jinja and chat
templates are not compiled in anyway), and keep upstream's `add_bos` formula verbatim. Net
effect equals b4531 behavior, every `use_jinja` branch stays consistent with what is linked,
and there is less divergence to re-merge next upgrade.

### Latent double-free in the reference implementation (not copied)

The reference fork's `icpp_free_model()` calls `llama_model_free(*g_model)` while its static
`common_init_result_ptr` still **owns** that model. The next `load_model` reassigns the
static — freeing the same pointer a second time. Ours calls `g_llama_init.reset()` instead,
freeing model and context together exactly once, then nulls the pointers and frees the
backend. Worth reporting to the authors.

### Behavior change to smoke-test

`common_init_from_params` used to run on every call, so `-c`, `-b` and seed from *that* call
shaped the context. With a static `g_llama_init`, the context is built from the **first**
call's args and later calls silently reuse it. Add a test that varies `-c` between
`load_model` and `run_update`.

## The native build is a STRICTER check than the wasm build (for LINK errors)

> **⚠️ SCOPE NOTE (added after local deploy).** This heading is true only for *link/exclusion*
> errors — native's lack of LTO retains paths the wasm build dead-strips. It is NOT a general
> claim that native is the stricter gate. The opposite is true for *runtime behavior*: the
> real WASM canister exposed FIVE bugs that native (111/111) passed clean — getenv faults,
> `use_mmap`, `warmup`, ggml static-init getenv, plus the dfx-cache confound. **Both gates are
> mandatory: `make all-tests` (native + wasm build) AND a local canister deploy that actually
> runs `load_model` + `run_update`.**

`icpp build-wasm` was green while **four** separate dependency errors sat latent. They only
appeared in `icpp build-native`, because the wasm build uses `-flto` and dead-strips
unreferenced paths that the native build retains:

| Missing | Referenced from | Why wasm hid it |
|---|---|---|
| `common/fit.cpp` | `common.cpp`, `sampling.cpp` | — (this one failed on wasm too) |
| `common/peg-parser.cpp` | `preset.cpp::parse_ini_from_file` (via `arg.cpp` -> `preset.h`) | LTO dead-stripped the ini path |
| `src/llama-model-saver.cpp` | `llama.cpp::llama_model_save_to_file` | LTO dead-stripped it |
| `ggml-cpu/arch/x86/*` | `ggml-cpu.c::type_traits_cpu` | wasm correctly uses `arch/wasm/` |

**Implication: a green `build-wasm` is NOT sufficient evidence that the exclusion list is
correct.** Those dependencies stay latent; a future upstream change that causes LTO to retain
one of those paths surfaces as a link failure with no obvious connection to today's exclusion
decision. **Gate releases on `make all-tests` (which runs both), never on `build-wasm` alone.**

Note the reference fork only ever built the WASM, so its far more aggressive exclusion list
may carry latent versions of exactly these.

### The `arch/` split has three separate consequences

b10076 moved per-architecture CPU kernels out of `ggml-cpu-quants.c` into
`ggml-cpu/arch/<arch>/`. That single refactor caused:

1. The missing `ggml_vec_dot_q8_0_q8_0` WASM SIMD kernel that motivated this whole upgrade.
2. The wasm build needing `arch/wasm/quants.c` added to `c_paths`.
3. The **native** build needing `arch/x86/{quants.c,repack.cpp,cpu-feats.cpp}`, or
   `type_traits_cpu` has undefined vec_dot entries (`mxfp4`, `nvfp4`, `q1_0`, `tq1_0`, `tq2_0`).

`[build-native]`'s arch entries are **x86_64-specific**. On an arm64 host, switch to
`arch/arm/`.

### ICPP-PATCH: `arch/wasm/quants.c` made inert off-WASM

icpp-pro **merges** `[build-wasm]` `c_paths` into the native build, so `arch/wasm/quants.c`
is compiled for the host too. Upstream guards only the SIMD kernel *bodies* with
`__wasm_simd128__`, leaving the function definitions to collide with `arch/x86/quants.c`
(13 duplicate symbols). Patched to wrap the whole file in `#ifdef __wasm__`.

### WASI stubs: guard some, not others

- `src/wasi-exception-stubs.cpp`, `src/wasi-dl-stubs.cpp` -> **`#ifdef __wasi__`**. They
  substitute for platform ABI (`__cxa_throw` in libc++, `dlopen` in libSystem) that the host
  provides for real; unguarded they produce duplicate symbols, and worse, would make the
  native MockIC binary trap on throw instead of unwinding.
- `src/wasi-download-stubs.cpp`, `src/wasi-chat-stubs.cpp`, `src/wasi-speculative-stubs.cpp`
  -> **unguarded**. They substitute for llama.cpp sources excluded from BOTH targets.

Getting this backwards breaks a build in either direction.

## R2 mitigation: prompt-cache format stamp (IMPLEMENTED)

**Confirmed as a live hazard, not a theoretical one.** The repo's checked-in b4531
`prompt.cache` has magic `ggsn` + version **9**; b10076 expects magic `GGSN` + version **9**.
The stale file passes both checks. Meanwhile the serialization definitively changed: b4531's
KV cache is a C-style struct with free functions and **zero** stream concept, while b10076's
`llama_kv_cache::state_read_meta(io, uint32_t strm, uint32_t cell_count, slot_info &, ...)`
gained a per-stream dimension from the `llama_memory_*` refactor. So an existing deployed
cache is **accepted and misparsed, silently**.

Mitigation, entirely in OUR code (zero added divergence in the vendored fork):

- `src/promptcache.{h,cpp}` — `PROMPT_CACHE_FORMAT` constant, sidecar `<cache>.icppfmt`,
  `prompt_cache_format_is_current()`, `prompt_cache_write_format_stamp()`,
  `prompt_cache_discard_if_stale()`.
- `src/run.cpp` — discards a stale cache before reuse, stamps new ones at creation.

The check is **positive**: a cache is trusted only if it carries a matching stamp. Unstamped
means "written by a build predating stamping", which is exactly the dangerous case; an
absence-based check would have silently trusted every legacy cache.

**Bump `PROMPT_CACHE_FORMAT` whenever a future llama.cpp upgrade changes session
serialization.**

**RELEASE NOTE REQUIRED:** upgrading a live canister invalidates existing prompt caches. They
are now discarded automatically, but users will see one slower first call per session.

## C4/C8 native runtime — inference WORKS, exact tokens UNCHANGED

Getting the native suite to green surfaced six runtime issues the builds could not (each
found and fixed in turn):

1. **Shims leaked into the native build.** icpp-pro merges `[build-wasm]` `cpp_include_dirs`
   into `build-native`, so `wasi-shims/` was first on the native path too. The native binary
   got the no-op mutex and the *trapping* `std::thread` ctor -> SIGILL (exit 132) when the
   log worker spawned. Fix: every shim now wraps its body in
   `#ifndef __wasi__ / #include_next <real> / #else / ...shim... / #endif`, plus the same for
   the `__mutex/*.h` sub-headers (with `#include_next` BEFORE the include-guard `#define`, or
   the real header self-suppresses).
2. **`common_download_run_tasks` trapped on the local-load path.** b10076 routes model
   resolution through the download machinery even for a local file; it calls `run_tasks`
   with an EMPTY task list. Fix: the stub returns early on an empty list; only a real
   (non-empty) download request traps. (Also set `params.offline = true` in `main_.cpp` as
   defense-in-depth, though the call happens at parse time, before `main_` runs.)
3. **`--model is required` — the axis mistake.** b10076 added a parse-time throw when
   `--model` is absent. The canister loads the model once and reuses it, so `new_chat`/`run`
   legitimately omit it. First guarded with `#ifndef __wasi__` -- WRONG: the native MockIC
   build does not define `__wasi__`, so the tests still failed. The correct axis is
   **canister-semantics vs real-CLI, not wasm vs native**. Added `-DLLAMA_CPP_CANISTER` to
   BOTH build sections and guarded with `#ifndef LLAMA_CPP_CANISTER`.

**Two guard axes — do not conflate:**
- `__wasi__` — platform. Correct when native has a working host equivalent (threads, `$HOME`,
  `<sys/syslimits.h>`). Native falls through to the real implementation.
- `LLAMA_CPP_CANISTER` — behavior. Correct when BOTH builds must act like a canister and
  there is no host fallback (the `--model` check). A wrong `#ifdef __wasi__` here would pass
  native tests and make the **mainnet canister reject every new_chat** — a defect that ships
  green. The native inference tests are what catch it; keep them as a release gate.

Audited the other four `__wasi__` sites (`set_process_priority`, `fs_get_cache_directory`,
syslimits, `log.cpp` threading) — all correctly on the platform axis (native uses a real
working implementation).

### RESULT: exact-token tests PASS unchanged (R1 did NOT materialize)

Every `test_tiny_stories` `run_update` — the ones asserting **verbatim generated text at

## C7 local deploy — install-time trap investigation

> **⚠️ READ FIRST — this whole investigation chased a phantom.** The install-time IC0502 was
> ultimately a **dfx wasm-CACHE artifact** (dfx installed a stale `.dfx/` copy, not the
> current build). It was NOT stack size, NOT `--initial-memory`, NOT the wasm content. Bug #1
> below (ggml static-init getenv) was a real bug found along the way, but it is subsumed by the
> global getenv override (Bug #5). Every hypothesis between here and "Bug #2 ACTUAL ROOT CAUSE"
> (`--initial-memory`, ic0 values, export count, stack overflow) was WRONG. Skip to
> **"Bug #2 ACTUAL ROOT CAUSE: dfx wasm caching"** for the resolution; the intervening text is
> kept to show the (long) debugging path and the reusable harness it produced.

The canister traps `heap out of bounds` (IC0502) at **install** on the local replica, before
any endpoint runs. Native (111/111) could not catch this — it is specific to the WASM
canister runtime. This is exactly why local deploy is a required gate.

### Debugging tools built (reusable)

Wrote a **faithful wasmtime harness** (`scripts/wasm_harness.py`): stubs the `ic0` imports,
backing stable memory with a real Python bytearray so the ic-wasi-polyfill init behaves as on
the IC. Runs against both the pre-wasi2ic reactor (`_initialize`) and the post-wasi2ic
canister (`build/llama_cpp.wasm`, start section at instantiation). This is the only way to get
a real backtrace for an install trap without a replica.

### Bug #1 FOUND AND FIXED: static-init getenv

The faithful harness gave the backtrace:
```
0: getenv
1: _GLOBAL__sub_I_ggml.cpp        <- ggml.cpp static initializer
2: __wasm_call_ctors
3: _initialize
```
`ggml.cpp`'s `ggml_uncaught_exception_init` (a file-scope `static bool = []{...}()`) calls
`getenv("GGML_NO_BACKTRACE")` at C++ static-init time. **Static-init order across TUs is
unspecified**, so it can run before icpp's `__wasi_polyfill` global ctor initializes the
polyfill; getenv then reads uninitialized polyfill state and FAULTS instead of returning NULL.

At this point I concluded: runtime getenv is safe (empty environ -> NULL), only
**static-init getenv is not** (polyfill not up yet). Guarded with `#ifdef __wasi__`.
> **⚠️ This intermediate conclusion was STILL WRONG** — runtime getenv also faults (Bug #5).
> getenv is unsafe *everywhere* under the polyfill, not just at static-init. This ggml guard
> is now redundant with the global override in `src/wasi-env-stubs.cpp` (left in place because
> it also disables a useless `std::set_terminate` backtrace handler). The real lesson is #5,
> not "static-init is the special case".

After this fix the faithful harness runs BOTH binaries' init clean (513 stable pages) — but
note that "clean init" only meant the *static-init* getenv was gone; the runtime one (Bug #5)
was still lurking and surfaced later at `load_model`.

### Bug #2: STILL trapping on the real replica, harness passes

With bug #1 fixed, the faithful harness passes the exact post-wasi2ic `build/llama_cpp.wasm`,
but the **real replica still traps IC0502 at install** (`dfx canister logs` shows only
`[TRAP]: heap out of bounds`, no debug output before it). So a second, IC-specific cause
remains that wasmtime does not reproduce.

Leading hypothesis: early-init code reads an `ic0` value (arg/caller/self size, time) and uses
it as a memory offset/length. The harness stubs those to 0 (safe); the real IC returns real
values. Note the production b4531 canister works WITHOUT `--initial-memory`, and the very
first trap predated that flag, so `--initial-memory` was never the cause -- reverted it.

### Bug #2 status: RULED OUT (not the cause)

- **`--initial-memory`**: tested WITH (512MB) and WITHOUT -> both IC0502. Not it. Reverted
  (b4531 known-good has no such flag).
- **ic0 return values**: faithful harness with REALISTIC ic0 (real time, self, caller, install
  args) still instantiates clean. Not an ic0-value-dependent access.
- **Static init**: the only file-scope static-init object that touches the polyfill was
  ggml.cpp's (bug #1, fixed). Model arch files (`src/models/*.cpp`) are pure method
  definitions -- NO static init -- so their count (137 vs Comoto's 17) is irrelevant to an
  install-time trap. peg-parser.cpp/preset.cpp have only static *functions*, no static-init
  objects.

The faithful wasmtime harness passes the EXACT post-wasi2ic binary the replica installs
(start section runs the ctors clean, 513 stable pages). The real replica still traps IC0502.
This is a fundamental wasmtime-vs-IC-embedder divergence that inspection cannot resolve.

Module metrics (current build, no --initial-memory): size 6.5MB, mem_min 9MB, functions 8078,
**exports 3567**, table 4345. The 3567 exports is abnormally high (a canister needs ~40) --
`--export-dynamic` leaking symbols; `optimize_wasm.py` strips exported GLOBALS but not
exported FUNCTIONS. Unclear if this is the cause (export count is usually a validation error,
not a runtime IC0502).

### Bug #2 ACTUAL ROOT CAUSE: dfx wasm caching (NOT stack, NOT the wasm)

**The b10076 build was fine all along. The install wall was a dfx caching artifact.**

dfx caches the module at `.dfx/local/canisters/llama_cpp/llama_cpp.wasm` and
`dfx canister install` uses THAT copy, not `build/llama_cpp.wasm`. During debugging I
hand-swapped `build/llama_cpp.wasm` many times (noopt 6994145 B, b4531 release, optimized
6564111 B) and ran repeated `install --mode reinstall`. dfx kept serving a STALE cached
binary -- the 1049-global no-optimize wasm -- so every install reported a trap/validation
error that had nothing to do with the current build.

Proof: `BinaryenGetNumGlobals(build/llama_cpp.wasm)` = **1 defined global**, but dfx install
reported "1049 globals" (IC0505) -- it was installing the stale `.dfx` copy. Installing with
explicit `--wasm build/llama_cpp.wasm` bypassed the cache and **succeeded**: module hash
`0x1f1df706...`, health -> `Ok {status_code=200}`, ready -> `Err "Model not yet loaded"`.

**Lessons:**
- Always install with `dfx deploy` (which recopies) or `dfx canister install --wasm <path>`.
  Never trust `install --mode reinstall` to pick up a hand-swapped `build/` wasm.
- `optimize_wasm.py`'s "Globals after optimization: 1" counts EXPORTED globals. The IC's 1000
  limit is on DEFINED globals; binaryen's `optimize()` genuinely reduces those to 1. Both are
  fine -- the 1049 was the stale noopt copy.
- The faithful wasmtime harness was RIGHT: the actual build instantiates clean. It could not
  reproduce the "trap" because there was no real trap in the current build -- dfx was serving
  a different binary.
- Time sink: the dfx cache confound sat under a genuine first bug (ggml static-init getenv,
  fixed) and made every subsequent test lie. Check what dfx actually installs (module hash /
  `.dfx` wasm size) before theorizing about the binary.

### Bug #3 (runtime, load_model): use_mmap patch was LOST in the merge

After the canister installed and the model uploaded correctly (sha256 verified matching the
smoke-test expected hash), `load_model` trapped IC0502 "heap out of bounds". Root cause:
b4531's `common/common.h` patched `use_mmap = false` ("not in a canister..."); the b10076
tree reverted to upstream's `use_mmap = true`, and we never pass `--no-mmap`. A canister has
no host filesystem to mmap -- the model is in the icpp virtual FS (stable memory) -- so the
mmap faults.

Native (macOS) passed because host mmap works there. Same native-vs-wasm axis as the other
patches, but this one is a real behavioral patch that the merge dropped. The merge agent's
TODO had flagged `use_mmap`, but it was conflated with `use_jinja` and never applied.

**Fix:** re-apply `use_mmap = false` in `common/common.h` (8th ICPP-PATCH).

### Bug #4 (runtime, load_model): warmup patch ALSO lost -- same class

With `use_mmap=false` applied, `load_model` STILL trapped IC0502. Second lost b4531 patch:
`common/common.h` `warmup = true` (upstream) must be **`false`** in a canister. Warmup runs a
`llama_decode` (empty batch) inside `common_init_from_params`; in the canister that decode
faults, and it wastes instructions. `wasi-exception-stubs.cpp`'s own header even documents
"params.warmup = false ... throws during llama_decode" -- the reminder was there, the patch
was not. (9th ICPP-PATCH.)

**Both use_mmap and warmup are Theme-E semantic patches in `common.h` that the three-way
merge never carried** -- the merge operated on `main_.cpp`, and these live in the vendored
`common.h`, which is replaced wholesale by the upstream checkout. Swept b4531's `common.h`
for other lost patches: only these two (`DEFAULT_MODEL_PATH` is gone upstream). Lesson: after
replacing the vendored fork, diff the OLD tree's `common.h`/`common.cpp` ICPP-PATCH markers
against the new tree and re-apply each -- do not rely on the main_.cpp merge to carry
common.h semantics.

### Bug #5 (runtime, load_model): getenv FAULTS -- Theme C was wrong

Named backtrace (from the faithful harness run on `before_opt`, which retains the wasm name
section that binaryen's optimize() strips):
```
0: getenv                <- faults at 0xffffffff
1: tty_can_use_colors()
2: common_params_parser_init(...)
3: common_params_parse(...)
4: canister_update load_model
```
`common_params_parse` -> `tty_can_use_colors()` -> `getenv("NO_COLOR"/"TERM")`, and **getenv
itself faults** -- under the ic-wasi-polyfill the libc `environ` pointer is garbage, so the
real getenv dereferences 0xffffffff.

**This corrects the Theme-C decision.** I had concluded runtime getenv "returns NULL safely
(empty environ)" and skipped all ~12 getenv guards. WRONG -- getenv traps. The reference
fork's stub comment said exactly this ("getenv() calls guarded to prevent garbage pointer
dereference"); I misread it as unnecessary.

**Fix (better than per-site guards):** `src/wasi-env-stubs.cpp` defines a strong `getenv`
(returning nullptr) in an always-linked TU. The linker satisfies the reference from ours and
never pulls wasi-libc's `getenv.o`, so EVERY getenv call site across llama.cpp/ggml is safe
at once -- no per-site `#ifdef` needed. (The ggml static-init guard from bug #1 becomes
redundant but is left in place; it also disables a useless terminate handler.)

This was also the true cause of the install-time getenv fault (bug #1) and likely would have
surfaced at many more sites; the override fixes them all.

### The stack-overflow hypothesis — ⚠️ WRONG (it was the dfx cache)

> **⚠️ CORRECTION.** When the install trap persisted, the git history (commit `6b24539`,
> README Appendix B) suggested "heap out of bounds = stack overflow in disguise, increase the
> stack." That precedent is real for the `load_model` GGUF-parsing case, but it was the WRONG
> diagnosis here: bumping `-Wl,-z,stack-size` from 8MB to **32MB did NOT fix the install
> trap** — the trap was the dfx wasm-cache artifact (next section). The reasoning that
> "wasmtime's stack is more generous than the IC's" was also wrong: the faithful harness
> passed because the *current build* genuinely instantiates clean, not because of a stack
> difference.
>
> **RESOLVED:** `icpp.toml` was REVERTED to `-Wl,-z,stack-size=8388608` (8MB, same as b4531)
> and re-validated — install + load_model + run_update all Ok on the local replica, mem_min
> back to 9MB. 32MB was never needed. Do not cite 32MB as "the install fix".
